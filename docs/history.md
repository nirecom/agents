### FEATURE: settings.json SSH-path deny pattern false-positive on command body text (2026-05-22, 58cb407daccf7b393261363e1e525f33bd8bb9c7, #424)
Background: 2026-05-21 のセッションで agents repo (cwd) から `gh issue create --repo nirecom/my-private-repo` を発行した際、ユーザーへの permission ダイアログ表示なしで `Permission to use Bash ... has been denied.` が返却された。 原因調査の結果、本 repo の `settings.json` line 204 にある以下の deny パターンが false-positive を起こしていることが判明: ``` "Bash(*~/.ssh/*)" ``` このパターンは substring glob のため、コマンド文字列全体 (`--body` argument や heredoc の中身を含む) に対して一致判定が行われる。issue body に `~/.ssh/config` 等の文字列が **説明用途で** 含まれているだけで巻き添え auto-deny される。 ## Symptoms / Evidence セッションログ `0f58b662-ee5d-4bbf-bd31-6ea0699ae16b.jsonl` の line 405〜413 のタイムスタンプ: | Time (UTC) | Event | |---|---| | 14:16:06.631 | assistant tool_use 発行 (`gh issue create --repo nirecom/my-private-repo --body "$(cat <<'EOF'...`) | | 14:16:06.797 | hook_success: `scan-outbound.js` | | 14:16:06.821 | hook_success: `block-dotenv.js` | | 14:16:06.822 | hook_success: `enforce-worktree.js` | | 14:16:06.822 | hook_success: `enforce-issue-close.js` | | 14:16:06.824 | hook_success: `enforce-system-ops.js` | | 14:16:06.832 | tool_result: `has been denied` | フック通過から ~8ms 後の deny。**ユーザーがダイアログで Deny を押す時間スケールではない**ため、Claude Code built-in の `permissions.deny` ルール照合による auto-deny で確定。 ## Root cause 意図された保護対象は `cat ~/.ssh/id_rsa` のような **SSH 秘密鍵への直接 read** だが、実装はコマンド文字列への substring 一致なので、`~/.ssh/` というリテラルがコマンド本文中にどんなコンテキストで含まれていてもマッチしてしまう。実際の file アクセス vs 単なる文字列言及を区別できない。 既存の以下 2 ルールは tool-input file path への一致なので適切に動作し、影響を受けない (本 issue の scope 外): ``` "Read(~/.ssh/**)" "Edit(~/.ssh/**)" ``` ## Proposed fixes ### 案 A (推奨): read コマンドに絞ったパターン群に置換 substring 一致は維持するが、対象を file-read/write 系コマンドの先頭に限定する。 置換前: ``` "Bash(*~/.ssh/*)" ``` 置換後 (例): ``` "Bash(cat *~/.ssh/*)" "Bash(less *~/.ssh/*)" "Bash(more *~/.ssh/*)" "Bash(head *~/.ssh/*)" "Bash(tail *~/.ssh/*)" "Bash(cp *~/.ssh/*)" "Bash(mv *~/.ssh/*)" "Bash(scp *~/.ssh/*)" "Bash(Get-Content *~/.ssh/*)" "Bash(Copy-Item *~/.ssh/*)" ``` -
